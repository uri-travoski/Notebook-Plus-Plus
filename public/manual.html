Installing with Docker Compose

What you need. A host with Docker and Docker Compose. The app image is public on GitHub Container Registry (GHCR), so no login or token is needed to pull it.

1. Get the compose file. Put docker-compose.standalone.yml (from the repository) in an empty folder. It defines the whole stack — the app plus its PostgreSQL database — so no other files are needed.

2. Generate three secrets. Each command below prints one random value (OpenSSL ships with macOS/Linux; on Windows use Git Bash or WSL):

openssl rand -hex 16     # -> DB_PASSWORD
openssl rand -base64 48  # -> SESSION_PASSWORD  (must be at least 32 characters)
openssl rand -hex 32     # -> ENCRYPTION_KEY

3. Create a .env file beside the compose file and paste the values in. Only the first three are required; the rest have sensible defaults:

DB_PASSWORD=<from openssl>
SESSION_PASSWORD=<from openssl>
ENCRYPTION_KEY=<from openssl>

# optional
APP_PORT=3000
ALLOW_REGISTRATION=true
NUXT_PUBLIC_APP_URL=https://notes.mywebsite.com
NUXT_SESSION_COOKIE_SECURE=false
SMTP_URL=

4. Start the stack:

docker compose -f docker-compose.standalone.yml up -d

5. Open the app at https://notes.mywebsite.com and register the first account — it arrives pre-loaded with the starter content. See Accounts & registration to turn sign-ups off afterwards.

Keep the three secrets safe and stable. Changing SESSION_PASSWORD logs everyone out; changing ENCRYPTION_KEY makes previously-saved AI keys undecryptable; changing DB_PASSWORD after the first run means the database must be recreated.

Data & backups. Everything lives in two Docker named volumes — notebookpp-pgdata (the database) and notebookpp-uploads (uploaded files). They survive restarts and image updates; back them up to back up the app.

Updating. Pull the newer image and recreate the containers — migrations run automatically at boot:

docker compose -f docker-compose.standalone.yml pull
docker compose -f docker-compose.standalone.yml up -d

To wipe everything and start clean, run docker compose -f docker-compose.standalone.yml down -v before up — this deletes both volumes.

HTTPS on a LAN. Over plain HTTP on a non-localhost IP, browsers disable some features (pasting into the canvas, installing the app as a PWA). To use those across your network, serve the app over HTTPS through a reverse proxy and set NUXT_SESSION_COOKIE_SECURE=true.

Accounts & registration

What it is. Notebook++ is self-hosted and account-based: each account is a private, independent workspace. Accounts are created by self-registration on the sign-up page (/register) — there is no shared or default login.

First run. A fresh install ships with no accounts. Open the app and register the first one; the first account on an empty install is always allowed, even when registration is otherwise disabled.

Starter content. Every new account is created pre-loaded with its own private copy of the starter content — a Getting Started notebook with sample notes (a formatting reference, a canvas, a data table, a Kanban board, and a short user guide), plus their tables and images. It is a full copy: edit or delete any of it freely without affecting other accounts.

Enabling / disabling registration. Whether new people can register is controlled by Settings → Registration (a toggle), which overrides the ALLOW_REGISTRATION environment default. Turn it off once your account(s) exist to keep the instance private.

Connections. The sign-up endpoint clones the bundled template into the new account (all ids and in-content references remapped, attachment files copied), then signs you in. The registration toggle is stored in the owner's preferences and read by the sign-up endpoint.

Database table block

What it is. An embedded, spreadsheet-style database you drop into any page note (via the / slash menu → "Database table"). Column data is stored relationally, so the same database can be queried outside the document.

Columns (properties). Each column has a name and a type: text, number, select, multi-select, date, checkbox, or url, each shown with a type icon in the header. Add a column with the + at the right edge of the header (name it and pick a type). Click a column header to open its menu: rename, change type, insert a column left or right, or delete it. Drag a column's right edge to resize it (double-click the edge to reset). For select/multi-select, options are created as you type them into a cell.

Views. A toolbar above the block switches between two views (the choice is saved per block):

  • Table — a Notion-style grid with a sticky header and a frozen first column. Click any cell to edit it inline; select cells open a colour-pill option picker; checkboxes and dates edit in place. Add rows with the + New footer. Hover a row to reveal its ⠿ grip — drag it to reorder rows, or click it for duplicate / delete. Scrolls in both directions for wide or long tables.
  • Board (Kanban) — groups rows into lanes by a Select column chosen in the "Group by" picker. Each row is a card showing its title (first text column) plus its other field values as chips. There is always a "No <column>" lane for rows with no value. Add a card to a lane with "+ New" (it presets that lane's value), and drag a card between lanes to change its Select value. The board scrolls horizontally.

Connections. Lives inside a page note's content; the editor island renders it but reads/writes rows through the /api/databases endpoints (the Vue shell owns document autosave). Needs at least one Select column to use Board view.

Backups & restore

What it is. A built-in, encrypted backup system (Settings → Backup) that captures your whole instance — the PostgreSQL database and your uploaded files — into a single encrypted .npbk file, on a schedule or on demand, to a local folder or any S3-compatible object store (Cloudflare R2, Backblaze B2, MinIO, AWS S3). Restores happen in place, with no downtime.

How a backup works. Each run:

  1. dumps the database with pg_dump (every note, canvas, database table, tag, and setting);
  2. archives the uploads folder (images, video, audio, file attachments) — when "Include uploads" is on;
  3. bundles them with a manifest and a copy of the instance ENCRYPTION_KEY;
  4. encrypts the whole bundle with AES-256-GCM using a key derived (scrypt) from your backup password; and
  5. uploads it to your destination, then prunes backups beyond your retention count.

Backups always nest under a notebookpp/ folder at the destination, so a shared bucket or drive stays tidy. Each file is named notebookpp-YYYYMMDD-HHMMSS.npbk.

Encryption. The backup password is not your login password, and it is not recoverable: without it a .npbk cannot be decrypted, by design. A small "check block" at the front of each file verifies the password in milliseconds before a restore touches anything, and GCM authentication makes a corrupted or tampered file fail to decrypt rather than restore garbage. Write the backup password down and keep it off this machine.

Set up backups (step by step).

  1. Open Settings → Backup.
  2. Set a backup password. Store it somewhere safe first — it encrypts every backup and cannot be recovered.
  3. Pick a destination:
    • Local folder — a directory inside the app container. In Docker, mount a host folder and enter its path; backups land in <dir>/notebookpp/.
    • S3 compatible — for Cloudflare R2: Endpoint https://<account-id>.r2.cloudflarestorage.com, Region auto, your Bucket, Force path style No (use Yes for MinIO/B2), and the Access key ID / Secret access key.
  4. Choose a schedule: Off (manual only), every 2 hours, every 6 hours, daily at 03:00, or weekly (Sun 03:00). Scheduled runs use the app's built-in job queue — no cron, no extra service.
  5. Set retention (default 14). Older backups are deleted after each successful run.
  6. Leave Include uploads on unless your uploads live in external object storage already.
  7. Click Save settings, then Back up now to prove the whole path works.

The Backups table lists each file with size and date; the History table logs every run. To verify a backup without changing anything, click Restore…, enter the password until it reads "Password OK", then Cancel.

Restoring. In Backups, click Restore…, enter the backup password (verified first), then type RESTORE to confirm. The app then, in place and without a restart: downloads and decrypts the backup, replaces the database (pg_restore --clean), restores the uploads folder (if included), runs any pending migrations, and re-keys your saved AI keys from the backup's embedded key to the running instance's key (so they keep working even if ENCRYPTION_KEY changed). This replaces everything not in the backup — anything created afterwards is lost. Reload the app when the banner clears.

Disaster recovery. If the server or its data is lost: bring the stack up empty, open Settings → Backup, re-enter your destination and backup password, then Restore… the newest file. Because the backup embeds your ENCRYPTION_KEY, restored AI keys keep working. The two things to keep off the machine are the backup password and (for S3) the bucket endpoint + keys: with those and any .npbk file you can rebuild the instance from nothing.

Gotchas.

SymptomCause & fix
"Set a backup password first."No password saved — enter one and Save settings.
"Wrong backup password."Use the password set when that backup ran — changing the password does not re-encrypt old backups.
Backup fails to S3Re-check the endpoint, bucket, and that the token has write access (R2: endpoint https://<account-id>.r2.cloudflarestorage.com, region auto, force-path-style No).
Backup fails to a local folderThe path must exist inside the app container — mount the host folder as a volume first.
Restored, but attachments are missingThe backup was taken with "Include uploads" off. Turn it on so uploads are captured.
A .npbk is lost with its passwordUnrecoverable by design. Keep the password off the machine and keep more than one backup (retention ≥ 2).

Connections. The panel talks to the /api/backup endpoints (all behind your login); scheduling uses the same in-app job queue as Markdown export; the password and S3 secret are encrypted at rest with the instance ENCRYPTION_KEY. Treat a .npbk as sensitive even though it is encrypted — it contains your whole instance.
